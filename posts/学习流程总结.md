# Web / SRC 信息搜集到 API 分析学习流程

> 适用场景：授权测试、靶场环境、自有资产梳理、SRC 学习。  
> 这份笔记不写正式漏洞报告部分，只整理**学习流程、工具定位、常用命令、API 分析思路和记录模板**。  
> 重点记住一句话：**前期是找资产，中期是筛重点，后期是找 API、看权限、看数据边界。**

## 1. 授权范围先确认

不要一上来就扫，先确认范围。

你要先知道：

```txt
哪些根域名在范围内？
哪些子域名在范围内？
哪些 IP 在范围内？
能不能扫端口？
能不能做子域名爆破？
能不能抓包测试接口？
能不能测上传 / 删除 / 修改？
```

注意：

```txt
扫到了 ≠ 可以测
同 IP ≠ 都在范围
旁站 ≠ 默认授权
历史解析 IP ≠ 一定属于当前目标
```

先准备一个根域名文件：

```txt
00_scope/root_domains.txt
```

示例：

```txt
example.edu.cn
example.com.cn
```

只放根域名，不要放 URL、路径、端口。

错误：

```txt
https://www.example.edu.cn/login
1.2.3.4:8080
www.example.edu.cn/admin
```

正确：

```txt
example.edu.cn
example.com.cn
```

### 2.3 httpx 安装

httpx 是 ProjectDiscovery 的 Web 探测工具，用来判断目标是否是 Web、拿标题、状态码、技术栈等。

Go 安装：

```bash
go install -v github.com/projectdiscovery/httpx/cmd/httpx@latest
```

Linux / macOS 加 PATH：

```bash
echo 'export PATH=$PATH:$HOME/go/bin' >> ~/.bashrc
source ~/.bashrc
```

验证：

```bash
httpx -version
httpx -h
```

Windows 如果用 Go 安装，通常在：

```txt
C:\Users\你的用户名\go\bin
```

需要把这个目录加到环境变量 `Path`。

### 2.7 Eeyes / EHole 说明

这里容易混：

```txt
Eeyes：偏向从大量域名里筛真实 IP，并整理 C 段。
EHole：偏向重点系统 Web 指纹识别。
```

你之前说的“放在 Eeyes 里看有没有 CDN 的 IP”，更接近：

```txt
用 Eeyes 帮助判断域名解析结果里哪些可能是真实 IP，哪些像 CDN。
```

Eeyes 常见参数：

```txt
-f    从 FOFA 搜索资产，支持 domain
-l    从本地文件读取目标
-log  指定日志文件
```

示例：

```bash
Eeyes -l domains.txt
```

如果配好了 FOFA：

```bash
Eeyes -f example.edu.cn
```

EHole 常见用法：

```bash
EHole -l url.txt
EHole -l url.txt -json export.json
```

注意：

```txt
Eeyes 不是漏洞扫描器。
它主要帮你从域名和 IP 里筛线索。
判断结果只能作为参考，还要结合历史解析、证书、httpx、人工访问一起看。
```

## 4. 被动信息搜集

被动信息搜集就是：

```txt
尽量不直接请求目标服务器，
先从公开平台、历史记录、证书、DNS 数据里找线索。
```

### 4.1 FOFA

常见查询：

```txt
domain="example.edu.cn"
host="example.edu.cn"
cert="example.edu.cn"
title="后台"
body="example.edu.cn"
```

组合：

```txt
domain="example.edu.cn" && title="后台"
domain="example.edu.cn" && port="8080"
cert="example.edu.cn"
```


```txt
URL
域名
IP
端口
标题
技术栈
是否像后台
是否有接口文档
是否是测试系统
```

## 6. 合并和清洗子域名

你可能有这些来源：

```txt
01_osint/fofa_domains.txt
01_osint/vt_domains.txt
01_osint/ct_domains.txt
01_osint/dnsdumpster_domains.txt
02_subdomains/oneforall_domains.txt
```

统一合并去重即可，不需要每个来源都单独去重。

PowerShell：

```powershell
Get-Content .\01_osint\fofa_domains.txt, `
            .\01_osint\vt_domains.txt, `
            .\01_osint\ct_domains.txt, `
            .\01_osint\dnsdumpster_domains.txt, `
            .\02_subdomains\oneforall_domains.txt |
Where-Object { $_.Trim() -ne "" } |
Sort-Object -Unique |
Set-Content .\02_subdomains\all_domains.txt
```

Linux：

```bash
cat 01_osint/fofa_domains.txt \
    01_osint/vt_domains.txt \
    01_osint/ct_domains.txt \
    01_osint/dnsdumpster_domains.txt \
    02_subdomains/oneforall_domains.txt \
  | sed '/^\s*$/d' \
  | sort -u \
  > 02_subdomains/all_domains.txt
```

最终文件：

```txt
02_subdomains/all_domains.txt
```

里面应该只有纯域名：

```txt
www.example.edu.cn
oa.example.edu.cn
api.example.edu.cn
```

不要有：

```txt
http://
https://
/login
:8080
标题
状态码
```

## 8. 提取 alive_domains 和 IP

### 8.1 提取可解析域名

PowerShell：

```powershell
Get-Content .\03_resolve\result.txt |
ForEach-Object { ($_ -split '\s+')[0] } |
Where-Object { $_.Trim() -ne "" } |
Sort-Object -Unique |
Set-Content .\03_resolve\alive_domains.txt
```

Linux：

```bash
awk '{print $1}' 03_resolve/result.txt | sort -u > 03_resolve/alive_domains.txt
```

### 8.2 提取 IP

PowerShell：

```powershell
Get-Content .\03_resolve\result.txt |
ForEach-Object { ($_ -split '\s+')[1] } |
Where-Object { $_ -match '^\d{1,3}(\.\d{1,3}){3}$' } |
Sort-Object -Unique |
Set-Content .\04_ip\ips.txt
```

Linux：

```bash
awk '{print $2}' 03_resolve/result.txt |
grep -E '^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+$' |
sort -u > 04_ip/ips.txt
```

## 9. CDN / 源站判断和源站确认思路

### 9.1 判断像不像 CDN

像 CDN 的情况：

```txt
同一个域名解析出很多 IP
不同地区解析结果不一样
IP 归属 CDN / 云防护厂商
CNAME 指向 CDN 域名
响应头里有 CDN / WAF 特征
访问 IP 本身不是业务页面
```

像真实业务 IP 的情况：

```txt
解析 IP 数量少
IP 归属比较固定
多个业务子域名解析到同一个 IP
访问 IP:端口 能看到默认站点、证书、后台、报错页
证书 SAN 里出现目标域名
```

---

### 9.2 Eeyes 辅助筛真实 IP

准备域名列表：

```txt
03_resolve/alive_domains.txt
```

运行：

```bash
Eeyes -l 03_resolve/alive_domains.txt
```

或者：

```bash
Eeyes -l domains.txt -log eeyes.log
```

如果配置 FOFA 后：

```bash
Eeyes -f example.edu.cn
```

Eeyes 结果要结合这些一起看：

```txt
DNS 解析结果
FOFA 证书搜索
VirusTotal 历史解析
微步历史解析
ip138 历史解析
httpx 访问结果
证书信息
页面标题
```

---

### 9.3 授权范围内的源站确认思路

如果判断某个域名走了 CDN，可以尝试找历史源站线索，但要注意：

```txt
找源站 IP 只是为了确认资产关系。
不能因为找到了历史 IP 就默认可以测试。
```

常见思路：

```txt
1. FOFA 查 cert="example.edu.cn"
2. FOFA 查 domain="example.edu.cn"
3. VirusTotal 看历史 resolutions
4. 微步看历史解析
5. ip138 看历史解析
6. 查证书 SAN 里还有哪些域名
7. 查邮件服务器 MX 对应 IP
8. 查较冷门子域名是否没接 CDN
9. 对疑似 IP 用 httpx 访问，看证书、标题、响应是否和目标相关
```

httpx 看证书和标题：

```bash
httpx -l 04_ip/ips.txt -title -status-code -tls-probe -follow-redirects -o 06_web/ip_web_check.txt
```

如果直接访问 IP 没内容，可以尝试在授权环境里用 Host 头确认站点绑定关系：

```bash
curl -k -H "Host: www.example.edu.cn" https://1.1.1.1/
```

注意：

```txt
这一步只做低频确认，不要对未知 IP 做大量请求。
如果 IP 不在授权范围，记录为线索即可。
```

```txt

```

---

### 11.2 基础探测

```bash
httpx -l 06_web/httpx_targets.txt -o 06_web/web_alive.txt
```

Windows：

```powershell
httpx -l .\06_web\httpx_targets.txt -o .\06_web\web_alive.txt
```

### 11.3 带标题、状态码、技术栈

```bash
httpx -l 06_web/httpx_targets.txt \
  -title \
  -status-code \
  -tech-detect \
  -follow-redirects \
  -o 06_web/web_info.txt
```

Windows 一行版：

```powershell
httpx -l .\06_web\httpx_targets.txt -title -status-code -tech-detect -follow-redirects -o .\06_web\web_info.txt
```

输出示例：

```txt
https://oa.example.edu.cn [200] [OA系统] [nginx,Spring Boot]
https://api.example.edu.cn [403] [Forbidden] [nginx]
http://1.1.1.1:8080 [200] [Apache Tomcat] [Tomcat]
```

状态码理解：

```txt
200：正常返回，重点看内容
301/302：跳转，重点看跳到哪里
401：需要认证，可能是后台或 Basic Auth
403：禁止访问，可能是权限、目录、WAF、默认拒绝
404：当前路径不存在，不代表整个站没用
500：服务异常，可能暴露技术栈
```

## 12. 指纹识别和重点资产筛选

可用工具：

```txt
httpx -tech-detect
Finger
EHole
Wappalyzer
Eeyes 辅助 IP 判断
浏览器插件
```

EHole 示例：

```bash
EHole -l 06_web/web_alive.txt
EHole -l 06_web/web_alive.txt -json 07_finger/ehole_result.json
```

## 19. GraphQL API 分析思路

GraphQL 和普通 REST API 不一样。

REST 常见是：

```txt
GET /api/user/info
GET /api/order/detail?id=1
POST /api/file/upload
```

GraphQL 常见是一个入口：

```txt
POST /graphql
POST /api/graphql
```

真正的查询内容在请求体里：

```json
{
  "query": "query { viewer { id name } }"
}
```

### 19.1 怎么判断是 GraphQL

看这些特征：

```txt
路径是 /graphql 或 /api/graphql
请求体里有 query
返回里有 data / errors
页面有 GraphiQL / GraphQL Playground
报错里出现 Cannot query field / GraphQL
```

---

### 19.2 Introspection 是什么

GraphQL 有 introspection 机制，可以查询 schema，也就是：

```txt
有哪些 Query
有哪些 Mutation
有哪些 Type
每个字段是什么
参数是什么
```

在授权测试里，可以先确认是否开放 introspection。

低风险查询示例：

```json
{
  "query": "{ __schema { queryType { name } mutationType { name } } }"
}
```

curl 示例：

```bash
curl -k -X POST "https://www.example.edu.cn/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __schema { queryType { name } mutationType { name } } }"}'
```

如果返回 schema 信息，说明可以继续整理 API 结构。

注意：

```txt
不要做深度过大的递归查询。
不要做批量查询压测。
不要构造消耗资源的复杂查询。
不要乱执行 mutation。
```

---

### 19.3 GraphQL 重点看什么

```txt
Query：查询类接口
Mutation：新增、修改、删除类接口
字段权限：普通用户能不能查不该查的字段
对象权限：A 用户能不能查 B 用户对象
敏感字段：手机号、邮箱、身份证、token、角色、权限
资源限制：有没有查询深度限制、复杂度限制
```

例如看到：

```txt
user(id: ID)
order(id: ID)
file(id: ID)
adminUsers
roles
permissions
```

就要想到：

```txt
id 能不能换？
普通用户能不能访问 adminUsers？
字段里有没有敏感信息？
mutation 是否需要高权限？
```

---


## 24. 常见 API 风险点

### 24.1 未授权访问

```txt
不登录也能访问接口。
```

例子：

```http
GET /api/user/info
```

如果未登录也返回用户信息，就是问题。

---

### 24.2 水平越权

```txt
A 用户能访问 B 用户的数据。
```

安全验证方式：

```txt
准备两个授权测试账号 A 和 B
A 创建自己的数据
B 创建自己的数据
用 A 的登录态访问 B 的数据 ID
```

不要做：

```txt
批量遍历 ID
批量下载文件
批量导出数据
```

---

### 24.3 垂直越权

```txt
普通用户能访问管理员接口。
```

例子：

```http
GET /api/system/user/list
GET /api/system/role/list
```

```txt
邮箱
学号
工号
家庭地址
token
密钥
角色权限
部门信息
内部 IP
数据库字段
```

### 24.5 文件下载问题

```txt
fileId
id
path
filename
url
```

看：

```txt
能不能下载别人的文件？
文件 ID 能不能换？
下载接口有没有鉴权？
```

### 24.6 文件上传问题

关注：

```txt

上传后文件能不能访问？
```

## 25. “数据爆破类”更准确的理解

这个词容易跑偏。

更准确叫：

```txt
参数枚举
对象 ID 对比
权限边界验证
数据范围验证
```

正确思路：

```txt
用两个授权测试账号做最小化验证。
```

例子：

```txt
A 的订单 ID：1001
B 的订单 ID：1002

用 A 的登录态访问 1002。
如果能看到 B 的订单，说明权限边界有问题。
```

不要做：

```txt
从 1 跑到 999999
批量下载
批量导出
撞密码
撞验证码
大量请求真实接口
```

一句话：

```txt
SRC 或授权测试里，要用最少请求证明权限边界，不是把数据全部跑出来。
```

## 28. 最终理解

你可以这样记：

```txt
FOFA / VT / 微步 / CT / DNSDumpster：
从公开来源找线索。

OneForAll：
集中收集子域名。

ksubdomain：
验证域名能不能解析。

Eeyes：
辅助筛真实 IP 和 C 段线索。

naabu / nmap：
看 IP 开了哪些端口。

httpx：
判断哪些目标是 Web。

Finger / EHole / Wappalyzer：
识别系统和技术栈。

截图：
人工挑重点资产。

Burp / Network：
从真实流量里找 API。

LinkFinder / JSFinder：
黑盒情况下从 JS 里挖 API 和 URL。

前端源码分析：
从配置、路由、请求封装、页面逻辑里还原 API。

Swagger / Knife4j / GraphQL：
从接口文档或 schema 里补全 API 地图。

App / 小程序：
从移动端补充 Web 上看不到的 API。

API 验证：
重点看未授权、越权、敏感字段、文件上传下载、权限边界。
```

最后一句：

```txt
前期工具负责把资产摊开，
中期截图和指纹负责筛重点，
后期 API 分析才是真正理解系统和发现问题的核心。
```
