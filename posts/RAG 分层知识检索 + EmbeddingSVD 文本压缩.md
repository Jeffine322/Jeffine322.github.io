# rag

rag检索增强技术，原理其实很简单，将文本切割成若干个chunk，然后计算出一个向量值。用户的query和chunk的向量值做余弦相似度的计算，返回相似度较高的top-k个。然后再把query和chunk过一次Transformer输出一个相关性分数，因为纯向量检索无法解决语义逻辑上的问题。



于是出现了两种模型Embedding和Rerank

Embedding是输入一段文本（query 或 chunk），输出一个固定维度的稠密向

Rerank输入 一对 (query, doc)，输出一个相关性分数，这个是过了一层attention的，效果是比单纯的文本相似度好很多。



由此展开了各种各样的优化方案，主要集中在query和chunk。但是只要不是过attention的相似度总会有问题，无论你用传统的无监督学习的那种方案，诸如TextRank，PageRank等等。



我在想能不能绕过Embedding直接Rerank所有的chunk呢？不现实，因为chunk如果上万，成本太高了，时间太久了。



那如果我改成树状的呢？树状的去做Rerank?但是问题来了，这个树应该如何去做，如果还是用文本相似度，那不还是Embedding模型干的事情吗？



现在的问题来了，怎么样更好的去制作这个树呢？



# 建树思路

那我们能不能这样子，先把相似的chunk聚集起来，然后使用llm为这个chunk生成一个浓缩的摘要？然后循环这个过程，来建树？并且需要是森林而不是单树，然后在这样子的树上rerank呢？这样子就不是单纯的Embedding去比较相似度了，比较的就比文本相似好很多。Embedding后聚类只是为了方便去做摘要。



我觉得是可行的，但是这个建树的过程可能很慢，但是新增知识没有问题，只需要更新某一个分支的所有节点就行了。成本也不低，普通人可能不一定烧的起一个庞大的知识库这子建树，但是我觉得这样子检索知识的效果一定会好很多



后面我发现这个思路就是RAPTOR（Recursive Abstractive Processing for Tree-Organized Retrieval） 的核心思想（2024年的论文）的核心思想。。。。原理原来这么简单，为什么轮不到我发这种论文

# 继续优化

首先这套方案做出来的效果应该是不错的，但是依然存在如下的问题

1.构建成本高、时间长

2.摘要的质量和成本，虽然Embedding后聚类就是为了提高摘要的成本，但是只要是摘要总会有信息的丢失



这两个问题也许能当成一个问题，如果能够优化摘要的成本和效果，这两个问题都能解决。我在想能不能不聚类后的多个chunk做SVD奇异值分解呢？因为文本肯定是有大量的废话的，能不能通过SVD分解把这些废话去掉？然后得到的摘要质量也会好一些，并且SVD奇异值分解的成本会比用llm低很多。



我觉得可行，来实现一个基于SVD奇异值分解的文本摘要算法吧





# 基于SVD奇异值分解的文本摘要算法

传统的无监督文本摘要算法的原理都很简单，对句子或词的频率，之间相似度来做取舍，来判断哪个句子重要然后保留，最终实现文本摘要。



这里可以用传统的TF-IDF值组成矩阵，例如列向量的句子，行向量是词，矩阵的元素值的含义就是，这个词在这个句子红的TF-IDF值(这个值的大致的作用是能区分这个词对全文的作用)。



然后奇异值分解，最终能得到一系列奇异值，根据奇异值能知道词的重要程度，但是现在要怎么拼回去得到一个摘要？如果要在摘要，那不就和原先的textrank和pagerank这种基于词频或者相似度打分的没有区别了吗？？因为我希望的是去掉废话，而不是将频率低的丢了，频率低的可能也很重要。



# embedding+SVD

那用embedding 把句子投影到稠密空间，再在 embedding 矩阵上做 SVD呢？

Embedding本身就在语义上有了一定的训练，以Word2Vec 代表的是预测为主，中间的权重层就是embedding。频率低但语义重要的内容已经被embedding拉近了。



这样子做的矩阵然后在SVD分解，这样子应该能更好地发现隐含语义主成分。



但是还是哪个问题，SVD后怎么得到摘要呢？SVD 选出 top K个句子靠谱吗？能不能有一种方法不是固定的选取K个句子，而且按照选取的K句子是否能几乎完整的代表整个文本呢



可以按照奇异值总和比上选取奇异值之和大于某一个值作为选取k的依据。得到了K个句子把这K个句子组合到一起能说明是摘要吗？

# 总结

也就是说这个思路单纯摘要效果其实没有那么理想，但依然可以使用，我觉得有如下两个思路可以去使用：

1.应该作为预处理的阶段，把长的chunk先用这种方式去提取主要的信息，然后再用原先的思路去做摘要，这样子能一定程度上降低成本，也能提升摘要的质量

2.因为是用还是先用embedding聚类，然后每一类用这个方式去摘要，大概率是能摘掉一些废话的，然后得到一批父节点，重复这个操作，直到摘要前后没有区别，然后再上llm，llm后再继续用这个方式摘要，最终得到一个森林。
