# 资产收集

> 主题：从一个域名 / IP / 单位名称出发，把公开线索串成资产关系图。  



怎么用公开信息去理解一个目标在互联网上留下了哪些痕迹。

资产收集 的核心不是工具本身，而是三个问题：

1. 这个线索从哪里来？
2. 这个线索能不能证明它属于目标？
3. 这个线索是强证据、中证据，还是弱线索？

简单定义：

> 资产收集，就是只使用公开渠道的信息，比如备案、DNS、证书、搜索引擎、公开代码、App 市场、公众号、小程序等，把一个目标背后的公开资产和关系慢慢串起来。

---

## 1. 总体思路

做 资产收集不要一开始就想着“扫到了多少东西”，而是要先建立一条主线：

```text
确认主体
↓
收集公开入口
↓
扩展域名 / 子域名 / App / 小程序 / H5
↓
分析 DNS / 证书 / 页面 / JS / 代码线索
↓
判断 IP / CDN / 云服务关系
↓
整理资产关系图
↓
标注证据来源和可信度
```

这里最重要的是“归属判断”。

查到一个子域名、一个 IP、一个接口地址，本身不代表它一定属于目标。  
只有当它能被备案、官网引用、App 主体、公众号、小程序、证书、页面内容等多个来源交叉印证时，才更有可信度。

---

## 2. 这是谁的？

很多人一上来就查子域名、端口、后台入口，但其实第一步应该先问：

- 这个域名到底是谁的？
- 这个系统是不是目标单位的？
- 这个 App 是谁开发的？
- 这个小程序主体是谁？
- 这个页面是官方入口，还是第三方外包平台？

可以看的公开来源：

ICP 备案、公安备案、WHOIS、官网页脚、App 市场开发者信息、小程序主体、公众号主体、隐私政策、用户协议、招标 / 中标公告、新闻稿、公开文档等

> 确认主体相当于给后面的资产扩展定边界。边界不清楚，后面查到的东西就容易误判。

 访问的页面是谁![image-20260606105151837](./assets/posts/资产收集/image-20260606105151837.png)

 进一步确认地址、电话、版权信息等确认主体![image-20260606105224665](./assets/posts/资产收集/image-20260606105224665.png)

## 4. 官网页脚

官网页脚经常被忽略，但它是确认主体和扩展资产的基础入口。

常见信息：

ICP 备案号、公安备案号、版权所有单位、技术支持单位、联系电话、联系邮箱、友情链接、公众号二维码、App 下载入口、隐私政策、用户协议、旧版入口、管理系统入口

为什么要看页脚？

因为页脚的信息通常是官方自己放出来的，可信度比搜索引擎结果更高。  
如果一个系统入口是官网直接引用的，它的归属可信度就会明显提高。

> 官网页脚像一个“公开导航”，里面经常能找到主体、备案、第三方厂商、业务入口和其他系统链接。

![image-20260606110215363](./assets/posts/资产收集/image-20260606110215363.png)

官网页脚列出和平校区、燕巢校区、学校总机、校安紧急通报、校区地图、交通资讯和版权信息。

## 5. 技术支持单位

很多官网底部会写：

- 技术支持：某某科技有限公司
- 建设单位：某某信息公司
- 运维单位：某某软件公司
- 平台支持：某某服务商

这类信息可以继续扩展：

- 这个厂商还服务过哪些单位？
- 其他单位是否使用同一套页面模板？
- 登录页是否相似？
- favicon 是否相同？
- 接口路径是否相似？
- 静态资源目录是否相似？
- 是否是常见建站系统或后台模板？

通过页面、图标、接口、目录、模板这些特征，判断不同系统是不是同一个厂商做的、是不是同一套系统改的 、如果是的话就可以分析他们之间是不是存在同源风险

> 同一个技术支持单位，只能说明可能存在建设或运维关系，不代表系统之间一定互通，也不代表资产一定属于目标单位。

这些特征其实就相当于系统的“长相”。比如登录页长得像、favicon 一样、接口路径差不多、静态资源目录也差不多，那基本就可以怀疑它们可能是同一个厂商做的一套系统，只是换了不同单位的名字和皮肤。

所以收集这些东西不是为了单纯看页面像不像，而是为了后面判断这个系统到底是哪来的，是不是某个厂商的产品，有没有其他单位也在用同一套。这样如果我在授权目标里发现了问题，就可以进一步判断：这是这个单位自己配置错了，还是这套系统本身就可能有通用问题。

后面再遇到类似的 favicon、登录页、接口路径、静态资源目录，也能很快反应过来：哦，这可能又是之前见过的那一套系统。

 从页面源码可以看到，该官网疑似使用 Joomla CMS，并引用了 JA Simpli 模板、Bootstrap、jQuery、SlideshowCK 等前端组件。![image-20260606110745098](./assets/posts/资产收集/image-20260606110745098.png)
这些信息可以作为被动指纹识别和技术栈判断线索

## 6. DNS：看当前解析和服务关系

DNS 是 信息收集里非常基础的一环。  
它主要告诉我们：域名现在指向哪里、由谁解析、邮件服务在哪里、有没有接入第三方服务。

常见记录：（可以理解为域名说明书）

| 记录类型 | 作用                                   | 可以看出什么                                               |
| -------- | -------------------------------------- | ---------------------------------------------------------- |
| A        | 把域名解析到 IPv4 地址                 | 当前域名指向的服务器 IP、是否可能是 CDN 节点               |
| AAAA     | 把域名解析到 IPv6 地址                 | 当前域名是否支持 IPv6、IPv6 服务器地址                     |
| CNAME    | 把一个域名指向另一个域名               | 是否接入 CDN、云服务、对象存储或第三方 SaaS                |
| NS       | 指定这个域名由哪个 DNS 服务器负责解析  | 域名使用的 DNS 服务商，比如阿里云、腾讯云、Cloudflare 等   |
| MX       | 指定这个域名的邮件服务器               | 邮箱服务商、邮件网关、企业邮箱使用情况                     |
| TXT      | 存放文本信息，常用于验证和安全配置     | SPF、DKIM、DMARC、站点验证、第三方平台验证信息             |
| SOA      | 域名区域的起始授权记录                 | DNS 区域的基础管理信息，比如主 DNS、管理员邮箱、刷新时间等 |
| PTR      | 反向解析，通过 IP 查域名               | 某个 IP 反向绑定的域名，常用于辅助判断服务器归属           |
| SRV      | 指定某个服务运行在哪台服务器、哪个端口 | 特定服务的主机、端口和协议，比如 SIP、LDAP、XMPP 等        |

常用命令：

```bash
nslookup example.com
nslookup -query=ns example.com 8.8.8.8
nslookup -query=mx example.com 8.8.8.8
nslookup -query=txt example.com 8.8.8.8
```

重点：

- A 记录只能说明当前解析到哪个 IP。
- CNAME 经常能看出 CDN 或第三方服务。
- MX 能判断邮件体系。
- TXT 能发现站点验证和办公平台痕迹。
- DNS 信息需要和备案、页面、证书继续交叉验证。

例如：

 主域名本身没有直接 A 记录返回，说明官网并不是直接挂在根域名上,而是放在 w3、sso、lis、news 这类子域上。![image-20260606112555293](./assets/posts/资产收集/image-20260606112555293.png)

DNS 不只是查 IP，它还能看出域名解析、邮件服务、第三方办公服务和安全策略线索。

下一步可以继续查一些业务子域：

```
nslookup w3.nknu.edu.tw
nslookup sso.nknu.edu.tw
nslookup lis.nknu.edu.tw
nslookup news.nknu.edu.tw
```

## 10. 空间测绘：看历史索引

空间测绘引擎可以理解成：

> 别人提前帮你把网上能访问到的服务、端口、标题、证书、指纹等信息收集好了。我们要做的就是用关键词去查，看看目标相关的资产有没有被它收录。

常见平台：

- Fofa
- Shodan
- ZoomEye
- Hunter
- Censys

常用查询思路：

```text
domain="example.com"
host="example.com"
cert="example.com"
icon_hash="..."
hostname:example.com
ssl.cert.subject.cn:example.com
http.favicon.hash:<hash>
```

能看到：域名、IP、端口、证书、标题、Header、favicon、服务指纹、历史暴露服务

注意：

> 空间测绘结果是历史索引，不一定实时。它适合发现线索，但不能直接当成最终结论。

例如  Fofa：![image-20260606150922430](./assets/posts/资产收集/image-20260606150922430.png)

其他的工具不演示了照着对应语法搜索就行

## 11. 子域名收集

子域名收集：边收集边判断它有没有价值，拿到域名之后可以思考以下问题

- 这个子域名是什么业务？
- 它是否还解析？
- 页面是否存在？
- 是否能证明属于目标？
- 是否是 CDN / 第三方平台？
- 是否是历史遗留？
- 是否是高价值入口？

来源：从不同公开来源把目标可能存在的入口找出来

比如：robots.txt、sitemap.xml、crossdomain.xml、crt.sh、Censys、搜索引擎、备案号、公司名、GitHub Search、grep.app、OneForAll、DNSDumpster等

高价值前缀：

| 前缀 | 可能含义 | 为什么关注 |
|---|---|---|
| api | 接口服务 | App / H5 / 小程序可能调用 |
| sso | 统一身份认证 | 能串起多个业务系统 |
| vpn | 远程接入 | 常见边界入口 |
| oa | 办公系统 | 可能涉及内部办公流程 |
| mail | 邮件系统 | 可和 MX 记录结合 |
| dev / test | 测试环境 | 可能是历史遗留 |
| m / h5 / app | 移动端业务 | 常见于活动页和小程序 |
| static / cdn | 静态资源 | 判断资源托管和第三方依赖 |
| admin / manage | 管理入口 | 只记录，不尝试登录 |

## 12. 证书透明度：发现历史子域

证书透明度平台可以查一个域名曾经申请过哪些证书。

常见平台：

- crt.sh
- Censys
- Entrust CT

示例：

```text
https://crt.sh/?q=%25.example.com
```

重点看：Common Name、SAN 字段、证书时间、证书颁发机构、是否出现其他子域名、是否出现历史域名、是否出现测试域名

> 证书里出现某个域名，只能说明它曾经被签进证书里。它是中证据，还要继续看 DNS、页面内容、备案、官网引用。

 示例：![image-20260606152709345](./assets/posts/资产收集/image-20260606152709345.png)

如上图就可以观察到目标域名下面曾经出现过一些子域名线索：

比如 `curriculum1-12.nknu.edu.tw` 和它的 `www` 版本，看名字可能和课程、教学系统有关；`fine-arts.nknu.edu.tw` 可能是艺术学院或美术相关站点；`ir-website.mail.nknu.edu.tw` 看起来和 mail 子域下的某个网站服务有关；另外还有 `kstssh108.nknu.edu.tw` 这种名字比较像服务器编号或主机名的记录等。

## 13. 被动指纹识别：判断相似性

被动指纹识别就是尽量不主动打扰系统，只从公开页面、响应头、证书、搜索引擎索引里看线索。

常见指纹：

| 类型 | 看什么 | 能说明什么 |
|---|---|---|
| favicon hash | 图标 hash | 同系统、同模板、同厂商 |
| HTTP Header | Server、X-Powered-By、Set-Cookie、Location | Web 容器、框架、跳转关系 |
| 网页源码 | meta、JS 路径、注释、静态资源域名 | 前端框架、接口前缀、第三方服务 |
| 错误页面 | 404 / 500 页面风格 | 统一模板、技术栈 |
| TLS 证书 | SAN、组织字段、颁发者 | 域名和组织线索 |
| Cookie | JSESSIONID、PHPSESSID、ASP.NET_SessionId 等 | 后端语言或框架方向 |

打开目标页面之后可以观察页面标题、登录样式、logo、favction及版权信息这些特征 发现有类似的就可以留意一下他们是不是一个厂商的模板

看浏览器开发者工具网络请求，观察响应头如server、X-powered-By、Set-Cookie、Location这些字段，可能会看出web容器、后端语言、框架类型等

观察网页源码和静态资源路径，如：JS、CSS、图片目录、接口前缀等这里可能会有源码暴漏在前端框架中

还有错误页面什么的都可以看看最后结合TLS证书、搜索引擎什么的做交叉判断

注意：

> 指纹只能帮助判断相似性，不能单独证明资产归属。

## 14. Swagger / API 文档痕迹

有些系统在开发或上线时，会保留 Swagger、Knife4j、OpenAPI 这类接口文档页面。如果这些页面没有做访问控制，就可能直接暴露系统的接口结构、业务模块、接口路径、参数名称和认证方式。

常见路径：

```text
/swagger-ui.html
/swagger-ui/
/v2/api-docs
/v3/api-docs
/doc.html
/api-docs
/openapi.json
```

记录内容：是否存在公开接口文档、接口模块名称、业务系统名称、认证方式、接口域名、是否暴露测试接口说明

注意：

> 这里只做公开页面判断，不做接口攻击。发现公开接口文档时，重点记录风险和暴露范围。

开发者名称、官网、隐私政策、版本更新时间、应用截图、权限说明、包名、关联 App

接口域名、H5 地址、WebView 地址、SDK Key、测试环境、静态资源地址、上传下载接口、业务模块名称

APKLeaks、jadx、apktool、七麦数据、点点数据、应用商店等

## 18. 隐私政策：确认主体和第三方依赖

隐私政策里也经常能看到：

公司全称、联系邮箱、客服电话、数据处理方、第三方 SDK、App 官网、API 域名、H5 域名、数据存储地区、用户协议链接、第三方共享清单

适合判断：App 和官网是否同主体、App 使用了哪些第三方服务、是否存在新的业务域名、是否有小程序 / H5 入口、是否有数据处理外包方

## 19. 小程序 / H5 / 公众号菜单：隐藏在移动端的入口

很多单位的业务入口不一定放在官网首页，反而会藏在公众号菜单、小程序、活动 H5、App WebView、二维码页面或者短链接跳转里。官网可能只是展示信息，但移动端入口里经常会有真正的业务功能，比如服务大厅、在线办事、查询系统、报名入口、缴费入口、成绩查询、预约系统、活动页面、小程序跳转、用户中心等。

收集的时候可以从官网链接、公众号菜单、公众号文章二维码、活动页面、App 内嵌页面等地方找入口

流程：

1. 从官网、公众号菜单、文章二维码里找入口。
2. 看小程序介绍页、服务类目、开发者主体、隐私政策。
3. 抓包观察请求域名。
4. 记录接口域名和业务关系。
5. 和官网域名、备案主体、证书 SAN、App 接口域名做交叉验证。

重点关注:

| 记录项     | 主要看什么                                             |
| ---------- | ------------------------------------------------------ |
| 入口来源   | 官网、公众号菜单、二维码、H5、App WebView、短链接      |
| 入口功能   | 报名、缴费、查询、预约、服务大厅、用户中心             |
| 跳转域名   | 点击后实际跳到哪里                                     |
| 移动端域名 | 是否出现 `/h5/`、`/mobile/`、`/wx/`、`/wechat/` 等路径 |
| 小程序信息 | 主体、开发者、服务类目、隐私政策                       |
| 接口域名   | 页面或小程序请求的 API 域名                            |
| 归属判断   | 是否和官网、备案、证书、页面版权一致                   |
| 业务价值   | 是否涉及登录、查询、缴费、文件、个人信息等功能         |

这类入口的价值在于，它们经常能补充官网没有暴露出来的业务系统和接口域名

## 20. 统一身份认证：串业务系统

如果发现这些入口：sso、auth、cas、oauth、login、passport、idp等可以重点关注一下

这些一般都和统一身份认证、单点登录、账号中心有关。SSO 本身不只是一个登录页面，它往往连接着很多业务系统，比如办事大厅、教务系统、OA、门户、应用中心、移动端服务等。

观察它连接了哪些系统。

可以记录：

- 哪些系统接入统一登录
- 登录页跳转参数里是否暴露业务系统 URL
- 回调地址指向哪些子系统
- 页面上是否有应用列表
- 是否有办事大厅 / 应用中心
- 是否有移动端入口

> SSO 不是单独一个系统，它往往是很多业务系统的入口。通过统一认证页面，可以还原一部分业务系统关系。

## 24. IP 反查和 CDN

同 IP 很容易误判。做 IP 反查和 CDN 判断，主要是为了避免把不属于目标的资产误收进来。

一个 IP 上可能会挂很多域名，但这些域名不一定都属于同一个主体。它们可能确实是同一个单位的资产，也可能只是共用了同一台云服务器、虚拟主机、CDN 节点，或者使用了同一个外包平台。所以不能因为“同 IP”或者“同 C 段”就直接判断它们属于目标。

判断时可以观察：

- 这个 IP 属于运营商、云厂商、IDC，还是 CDN 节点
- 是否能看到 ASN、地理位置、教育网、政务云等归属信息；
- 多地 Ping 出来的 IP 是否一致；
- CNAME 是否指向 CDN 厂商；
- Header 里是否有 CDN 厂商痕迹；
- 证书、备案、页面内容是否能和目标主体对应；
- 页面版权、系统名称、登录页信息是否能证明归属。

这一步的价值在于辅助判断服务器托管位置、是否走 CDN、是否使用云服务，以及这个资产和目标之间的关系强不强。

> 同 IP 不等于同主体，同 C 段也不等于同资产

## 26. 资产优先级判断 + 薄弱点分析

整理完资产后，要先分优先级。

### 判断薄弱点的常见特征

#### 一、组件版本明显过旧

这是最直观的一类。比如截图里那种资产，能看到：

- IIS 6.0
- Apache 2.2.x
- PHP 5.x
- WordPress 4.x / 5.x
- Drupal 7.x
- Joomla 旧版本
- OpenSSL 旧版本
- 老旧 Nginx / Tomcat / WebLogic / ThinkPHP / Struts 等

这种资产不一定百分百有漏洞，但它说明两个问题：

1. 系统维护频率可能不高；
2. 公开历史漏洞的可能性更大。

| 证据来源            | 说明                                 |
| ------------------- | ------------------------------------ |
| Server 响应头       | 比如 Apache、Nginx、IIS、Tomcat 版本 |
| X-Powered-By        | 可能暴露 PHP、ASP.NET、Express 等    |
| HTML 源码           | 可能有 generator、CMS 版本、框架特征 |
| JS / CSS 文件名     | 可能暴露框架、构建时间、版本号       |
| 登录页特征          | 某些系统默认登录页非常明显           |
| 报错页面            | 可能泄露框架、路径、版本、语言       |
| favicon             | 可辅助识别系统类型                   |
| 页面底部版权        | 有时会显示年份、系统名称、厂商       |
| 公开文档 / 操作手册 | 可能写明系统版本或厂商产品名称       |

#### 二、默认页面、默认标题、默认路径

如果一个资产打开后还是默认页面，通常说明维护不够细致。

常见表现有：

| 特征                          | 说明                                          |
| ----------------------------- | --------------------------------------------- |
| Welcome to nginx              | Nginx 默认页，可能是临时环境或未配置完成      |
| Apache / Tomcat 默认页        | 可能存在管理后台、示例目录或默认配置          |
| XAMPP / phpStudy / 宝塔默认页 | 常见于临时部署或测试环境                      |
| 默认后台路径                  | 如 `/admin`、`/login`、`/manager`、`/console` |
| 默认错误页面                  | 暴露中间件、语言、路径结构                    |
| 默认 CMS 安装页               | 说明系统可能未初始化完成或配置不当            |

这类资产的价值在于：
 它不一定本身就是核心系统，但它可能暴露了目标单位的部署习惯、技术栈、测试环境、历史系统。

#### 三、系统看起来长期没人维护

这个判断很重要，尤其是学校、单位、部门站点里经常出现。

可以观察这些点：

| 特征                   | 可能说明                 |
| ---------------------- | ------------------------ |
| 页面版权停留在很多年前 | 系统长期未更新           |
| 新闻公告多年未维护     | 站点可能废弃但仍在线     |
| 页面样式很老           | 可能是旧 CMS 或旧框架    |
| HTTP 明文访问          | 安全配置可能较弱         |
| 证书过期或证书信息异常 | 运维维护不到位           |
| 页面存在大量报错       | 代码或环境长期无人修复   |
| 链接大量 404           | 系统迁移不完整           |
| 附件、说明文档还在     | 可能有历史路径和系统线索 |

这种资产要重点记录，因为它们经常不是“最核心”的入口，但可能是整条资产链里比较薄弱的一环。

#### 四、暴露了后台、接口或管理入口

这类资产优先级通常比较高。

重点关注：

| 暴露内容              | 关注原因                                 |
| --------------------- | ---------------------------------------- |
| 登录后台              | 可能是管理端、运维端、业务端             |
| API 接口              | 可能存在数据查询、身份校验、权限控制问题 |
| Swagger / API 文档    | 可能直接暴露接口结构                     |
| 文件上传入口          | 涉及附件、图片、文档流转                 |
| 文件下载入口          | 可能涉及资料、报表、附件                 |
| 数据查询页面          | 可能涉及人员、订单、统计、业务数据       |
| 统一认证入口          | 可能关联多个系统                         |
| VPN / 网关 / 远程接入 | 边界资产，风险更高                       |
| 监控 / 运维面板       | 可能暴露内部系统状态                     |

判断时可以这样想：

> 这个页面有没有“登录、查询、上传、下载、导出、审核、管理、配置、统计、接口调用”这些功能？

有的话，就比普通展示页更值得看。

#### 五、信息泄露痕迹明显

有些资产看起来只是普通页面，但它会泄露很多线索。

常见线索包括：

| 类型     | 示例                                         |
| -------- | -------------------------------------------- |
| 目录结构 | `/upload/`、`/file/`、`/backup/`、`/static/` |
| JS 接口  | 前端代码里暴露 API 地址、参数名              |
| 注释信息 | 开发人员备注、测试接口、旧地址               |
| 公开附件 | 操作手册、账号格式、系统说明                 |
| 报错信息 | 真实路径、数据库类型、框架信息               |
| 备份痕迹 | `.bak`、`.old`、`.zip`、`.rar`、`backup`     |
| 测试环境 | `test`、`dev`、`uat`、`demo`、`beta`         |
| 内网地址 | `10.x.x.x`、`172.16.x.x`、`192.168.x.x`      |

这些信息不一定直接是漏洞，但能帮助你继续扩展资产关系图。

### 高价值资产

| 资产类型           | 关注原因                                       |
| ------------------ | ---------------------------------------------- |
| SSO / 统一身份认证 | 通常连接多个业务系统，是资产关系图里的核心入口 |
| VPN                | 远程接入入口，通常属于边界资产                 |
| OA                 | 涉及办公流程、人员、审批等内部业务             |
| API 网关           | 可能集中承载多个系统接口                       |
| 管理后台           | 具有系统管理功能，权限价值较高                 |
| 文件上传 / 下载    | 可能涉及文件流转、附件、资料下载等功能         |
| 数据查询系统       | 可能承载业务数据、统计数据或人员数据           |
| 支付 / 缴费系统    | 涉及资金、订单、缴费记录等敏感业务             |
| 统一门户           | 通常聚合多个业务入口和用户身份体系             |
| App 接口           | 移动端常用接口，可能连接核心业务系统           |
| 小程序接口         | 常见于移动办事、查询、报名、缴费等场景         |
| 一网通办           | 聚合多个办事流程和业务系统                     |
| 数据中台           | 可能集中承载数据交换、统计、分析等能力         |

### 中价值资产

| 资产类型               | 关注原因                                   |
| ---------------------- | ------------------------------------------ |
| 官网                   | 主要用于主体确认、入口导航和公开信息收集   |
| 新闻站                 | 可发现公告、业务流程、系统名称和历史线索   |
| 二级部门站             | 可帮助梳理组织结构和下属单位入口           |
| 专题活动页             | 可能包含活动报名、通知、附件和临时业务入口 |
| 静态资源站             | 可用于分析资源引用、前端文件和第三方依赖   |
| 帮助中心               | 可能包含系统说明、操作流程和常见问题       |
| 操作手册               | 可还原系统入口、使用对象和业务流程         |
| 招生 / 招聘 / 通知页面 | 可发现公开业务流程、系统名称和联系方式     |

### 低价值资产

| 资产类型       | 关注原因                                     |
| -------------- | -------------------------------------------- |
| CDN 静态资源   | 多为资源分发节点，通常不能直接证明资产归属   |
| 纯展示页       | 业务交互少，主要用于公开展示                 |
| 第三方跳转页   | 属于外部平台或外部依赖，不一定是目标自有资产 |
| 过期历史页面   | 可作为历史线索，但需要确认是否仍然有效       |
| 搜索引擎缓存   | 可能过期，只能作为弱线索                     |
| 无业务功能页面 | 资产价值较低，通常只做记录和归类             |

资产优先级不是按域名好不好看分，而是按它承载的业务、数据和入口价值来分。
